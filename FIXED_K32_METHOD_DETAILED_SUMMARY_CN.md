# fixed_k32 方法说明

这份文档只讲清楚 `fixed_k32` 这个算法本身是怎么做的。除“已有实验效果”一节外，不展开比较其他方法。

一句话概括：`fixed_k32` 是一种**每一步固定选 32 个源点来近似边介数，然后在 v3 候选边集合中删除估计分数最高边**的网络瓦解算法。

## 1. 它要解决什么问题

这里的任务是边删除式网络瓦解。给定一张当前图，每一步删掉一条边，希望最大连通分量尽快变小。

**最大连通分量**，也就是 GCC，是当前图里节点数最多的连通块。GCC 越快下降，说明网络越快被瓦解。

一个经典想法是每一步计算所有边的**边介数**。边介数表示一条边出现在多少最短路径上。边介数高，说明这条边可能是图中信息流或连接通道的重要边。

完整计算边介数很贵，因为需要从很多源点出发做最短路径搜索。`fixed_k32` 的做法是：**不使用所有源点，只固定使用 32 个源点来近似候选边的边介数**。

## 2. 算法输入和输出

每一步输入：

- 当前图 `G_t`。
- 当前最大连通分量 `GCC_t`。
- 当前步数 `step`。
- 图编号 `graph_id`。
- 随机种子 `seed`。
- 固定源点预算 `k = 32`。

每一步输出：

- 本步要删除的边。
- 删除后的 GCC 大小和 GCC 比例。
- 本步使用的真实源点遍历次数：32。
- 如果有 full-M5 参考，还会记录所选边是否等于 full-M5 top1、所选边在 full-M5 排序中的名次等诊断指标。

## 3. 第一步：取当前 GCC

算法每一步先从当前图中取最大连通分量：

```text
H = 当前图 G_t 的最大连通分量
```

之后的候选边构造、社区划分、源点选择和边打分，全部在这个当前 GCC 上进行。

如果当前 GCC 节点太少，或者已经没有边，算法就无法继续正常选边。

## 4. 第二步：更新社区划分

`fixed_k32` 使用 v3 的候选边集合，而 v3 候选边需要知道当前图的社区结构。

代码中使用 Louvain 社区划分。Louvain 是一种把图划分成若干社区的算法。社区内边较密，社区间边较少。

注意：代码不是每一步都无条件重新跑 Louvain，而是用 v3 的漂移检测机制决定是否重算。触发条件包括：

- modularity 漂移阈值：0.05。
- cross-community edge ratio 漂移阈值：0.05。
- boundary 节点集合 Jaccard 阈值：0.70。
- source routing 比例 L1 漂移阈值：0.25。
- 综合 drift score 阈值：1.50。
- 最小重算间隔：`max(5, ceil(0.01 * m0))`。

如果旧社区划分已经不能覆盖当前 GCC 节点，会触发 coverage 例外重算。

## 5. 第三步：构造 v3 候选边集合

`fixed_k32` 不是在当前 GCC 的所有边里直接选边，而是先构造一个候选边集合。这样做的意义是：先用便宜的结构规则筛出更值得考虑的边，再用 32 个源点去估计这些边的边介数。

候选边构造函数是：

```text
build_candidate_edges_v3(graph, partition, structural, profile, state=None)
```

当前 prefix smoke v2 使用的是 v3 默认候选参数：

| 参数 | 数值 | 含义 |
|---|---:|---|
| `min_fraction` | 0.10 | 候选边数量至少是当前 GCC 边数的 10% |
| `max_fraction` | 0.50 | 候选边数量最多是当前 GCC 边数的 50% |
| `degree_top_fraction` | 0.10 | 额外保留 degree-product 排名前 10% 的边 |

候选边来自五类结构边的并集。

第一类是跨社区边。若一条边的两个端点属于不同 Louvain 社区，就加入候选集。这类边可能连接不同社区。

第二类是边界节点相关边。若一条边至少有一个端点是边界节点，就加入候选集。边界节点指连接到其他社区的节点。

第三类是真实桥边。桥边指删除后会增加连通分量数量的边。这类边天然可能切开图。

第四类是低共同邻居边。共同邻居数是边两端节点共同拥有的邻居数量。如果共同邻居数不超过 1，就认为这条边局部嵌入较低，更像局部桥。

第五类是高度数乘积边。按 `degree(u) * degree(v)` 从高到低排序，取前 10%。这类边可能连接高影响力节点。

候选并集写成：

```text
候选原始集合 =
    跨社区边
    ∪ 边界节点相关边
    ∪ 桥边
    ∪ 低共同邻居边
    ∪ 高 degree-product 边
```

如果原始候选太多，就按优先级截断到最多 50% 当前边数。优先级大致是：

```text
同时属于更多类别的边优先
桥边优先
跨社区边优先
边界节点相关边优先
低共同邻居边优先
degree-product 更大的边优先
```

如果候选太少，就用 degree-product 排名靠前的边补足到至少 10%。

最终得到当前步候选边集合：

```text
C_t = 当前步候选边集合
```

## 6. 第四步：把节点分成四类源点池

`fixed_k32` 的 32 个源点不是直接从全体节点均匀随机抽取。代码会先把当前 GCC 中的节点分成四类：

| 类别 | 含义 |
|---|---|
| `boundary` | 边界节点，连接到其他社区 |
| `core` | 核心节点，度数高于当前平均度 |
| `community` | 社区代表节点，每个社区选一个度数较高的代表 |
| `random` | 随机/剩余节点池 |

这一步对应代码函数：

```text
build_mutually_exclusive_strata(...)
```

四类节点互不重叠，并且合起来覆盖当前 GCC 的所有节点。

## 7. 第五步：从四类源点池中选 32 个源点

选源点分两步。

第一步，为每一类节点池生成一个可复现的随机顺序：

```text
seeded_pools(strata, graph_id, step, seed)
```

它会对 `boundary`、`core`、`community`、`random` 四类节点分别洗牌。洗牌使用稳定随机种子，所以同一个图、同一步、同一个 seed 下结果可复现。

第二步，按配额从四类节点池中取节点：

```text
allocate_sources(context, pools, target=32)
```

配额函数是：

```text
target_quotas(proportions, populations, target, existing_counts)
```

主要规则：

- 目标总数是 32。
- 每个非空类别尽量至少取 2 个节点。
- 单个类别最多约占 50%，也就是最多约 16 个。
- `random` 类别至少保留约 10%，对 32 个源点来说至少约 4 个。
- 如果某类节点不够，就从其他类别补足。

所以，`fixed_k32` 不是简单的全图均匀随机抽 32 个节点，而是**按 v3 的结构分层规则选 32 个源点**。

最终得到：

```text
S_32 = 当前步选出的 32 个源点
```

如果当前 GCC 节点数少于 32，实际源点数会小于 32。

## 8. 第六步：对每个源点计算单源边贡献

对每个源点 `s`，算法运行一次 Brandes 式单源最短路径计算。

Brandes 单源计算包含两部分。

第一部分是 BFS，从源点 `s` 出发，遍历当前 GCC，记录：

- 每个节点到 `s` 的距离；
- 从 `s` 到每个节点的最短路径条数；
- 最短路径前驱节点。

第二部分是反向依赖累积，从距离最远的节点往回累积贡献，把经过每条边的贡献加到该边上。

对一条候选边 `e`，源点 `s` 给它的贡献记为：

```text
contribution_s(e)
```

代码函数是：

```text
single_source_contribution(source, edge_index, adjacency, scale)
```

这里 `scale = max(1.0, 当前 GCC 节点数 - 1)`，所以单源贡献做了一个归一化。贡献值会被限制在 0 到 1 之间。

## 9. 第七步：给每条候选边打分

对候选边 `e`，`fixed_k32` 把 32 个源点对它的贡献取平均：

```text
score(e) = 32 个源点对边 e 的 contribution_s(e) 的平均值
```

代码里对应的是：

```python
matrix[np.asarray(rows, dtype=int), :].mean(axis=0)
```

这里没有显式乘上 `n_t / 32`。因为当前步只需要比较候选边之间的相对大小，乘不乘同一个常数不会改变排序。

## 10. 第八步：删除分数最高的候选边

所有候选边都有分数后，算法按分数从高到低排序：

```text
e_star = 候选边中 score 最大的边
```

然后删除这条边：

```text
G_{t+1} = G_t 删除 e_star 后的图
```

删除后重新计算 GCC 大小和 GCC 比例，进入下一步。

## 11. 完整伪代码

```text
输入：
    当前图 G_t
    图编号 graph_id
    当前步 step
    随机种子 seed
    固定源点数 k = 32

每一步：
    1. H = G_t 的最大连通分量

    2. partition = 根据 v3 漂移规则更新或复用 Louvain 社区划分

    3. C_t = 构造 v3 候选边集合
       候选边来自：
           跨社区边
           边界节点相关边
           桥边
           低共同邻居边
           高 degree-product 边

    4. strata = 把 H 中节点分成 boundary/core/community/random 四类

    5. pools = 对四类节点池分别生成可复现随机顺序

    6. S_32 = 按 v3 配额规则从四类节点池中选 32 个源点

    7. 对每个源点 s in S_32：
           运行一次单源 Brandes 计算
           得到每条候选边 e 的 contribution_s(e)

    8. 对每条候选边 e in C_t：
           score(e) = 32 个源点贡献的平均值

    9. e_star = score 最高的候选边

    10. 删除 e_star

    11. 记录删除后的 GCC 大小、GCC 比例、所选边、运行时间等
```

## 12. 真实计算成本

`fixed_k32` 每一步真实需要 32 次单源 Brandes 遍历。

如果当前 GCC 有 `n_t` 个节点、`m_t` 条边，那么一次单源 Brandes 大致遍历当前 GCC 的节点和边。因此每一步主要成本可以理解为：

```text
约 32 次完整的单源遍历
```

或更具体地写成：

```text
约 O(32 * (n_t + m_t))
```

这里的 32 是真实源点遍历次数，不是估算出来的有效成本。

当前 prefix smoke v2 的代码为了诊断，会先计算 full-M5 参考矩阵，也就是对所有节点都算一遍单源贡献。这个诊断预计算会让不同方法的运行时间看起来接近。因此，如果讨论真实部署成本，应看真实源点遍历次数，而不是那次诊断运行的 wall-clock runtime。

## 13. 关键参数表

| 参数 | 数值 | 作用 |
|---|---:|---|
| 固定源点数 | 32 | 每一步最多选 32 个源点 |
| 候选边最小比例 | 0.10 | 候选边至少占当前 GCC 边数 10% |
| 候选边最大比例 | 0.50 | 候选边最多占当前 GCC 边数 50% |
| degree-product 保留比例 | 0.10 | 保留 degree-product 前 10% 的边 |
| 源点类别 | 4 类 | boundary、core、community、random |
| 单类源点上限 | 约 50% | 防止某一类源点占比过高 |
| random 源点下限 | 约 10% | 保留随机探索 |
| Louvain modularity 漂移阈值 | 0.05 | 判断是否重算社区 |
| cross ratio 漂移阈值 | 0.05 | 判断跨社区结构是否变化 |
| boundary Jaccard 阈值 | 0.70 | 判断边界节点集合是否变化 |
| routing L1 漂移阈值 | 0.25 | 判断源点分配比例是否变化 |
| drift score 阈值 | 1.50 | 综合漂移触发阈值 |

## 14. 已有实验效果

这一节只引用已有结果，不重新跑实验。

在 prefix smoke v2 中，测试图为：

- `ca_netscience`
- `bio_diseasome`
- `ia_enron_only`

每个图运行 40 个前缀删除步骤，seed 为 20260513。

主要结果：

| 方法 | prefix AUC | active AUC | 40 步后 GCC 下降量 | 与 full-M5 top1 一致率 | 真实源点遍历次数 |
|---|---:|---:|---:|---:|---:|
| full-M5 参考 | 0.618579 | 0.564212 | 0.537111 | 1.000000 | 平均约 190.258 |
| `fixed_k32` | 0.597816 | 0.540659 | 0.579727 | 0.591667 | 32 |
| 固定 48 源点 | 0.599134 | 0.542857 | 0.558608 | 0.700000 | 48 |
| 两阶段采样方法 | 0.609018 | 0.553722 | 0.542365 | 0.700000 | 56 |

这里 AUC 越低越好，GCC 下降量越高越好。

可以看到，`fixed_k32` 的 active AUC 最低，40 步后 GCC 下降量最高。因此，在这组已有动态实验里，`fixed_k32` 是表现最好的少源点基线。

但要注意：它不是因为最接近 full-M5。它与 full-M5 top1 的一致率是 0.591667，低于一些更精细的估计方法。当前解释是：`fixed_k32` 的 32 源点采样有一定噪声，这种噪声有时会让它避开高介数但删不动 GCC 的核心内部边，转而选到更能切出外围节点的边。

在 `ia_enron_only` 这个图上，这个现象最明显：

| 方法 | active AUC | 40 步后 GCC 下降量 | 所选边的平均 full-M5 排名 |
|---|---:|---:|---:|
| full-M5 参考 | 0.882044 | 0.118881 | 1.000 |
| `fixed_k32` | 0.838749 | 0.209790 | 7.625 |
| 固定 48 源点 | 0.869704 | 0.132867 | 2.300 |
| 两阶段采样方法 | 0.869910 | 0.132867 | 2.775 |

这说明：在这个图上，排名最靠前的 full-M5 边不一定最能降低 GCC；`fixed_k32` 选到的边平均 full-M5 排名更低，但动态瓦解效果更好。

## 15. 这个算法的直观理解

可以把 `fixed_k32` 理解成三层结构。

第一层是结构筛选：先用 v3 候选边集合排除大量不太可能重要的边。

第二层是分层采样：从 boundary、core、community、random 四类节点中选 32 个源点。

第三层是边介数近似：用这 32 个源点的 Brandes 单源贡献平均值，估计候选边的重要性。

最后，算法删除估计分数最高的候选边。

它的特点是简单、固定、可复现，并且真实源点成本明确，就是每步 32 次单源遍历。

## 16. 局限性

`fixed_k32` 不是理论最优算法。

它有几个限制：

- 每一步固定 32 个源点，不能根据状态自动增减预算。
- 它不保证最接近 full-M5。
- 它依赖 seed；虽然可复现，但换 seed 可能改变具体选边。
- 它依赖 v3 候选边集合；如果候选集漏掉关键边，它也选不到。
- 它在当前已有动态实验中表现好，但仍需要更多图、更多 seed 验证。

## 17. 可以放进 PPT 的简短表述

`fixed_k32` 是当前已有动态实验中表现最好的少源点边介数基线。它每一步先在当前最大连通分量上构造 v3 候选边集合，再从 boundary、core、community、random 四类节点中按配额选择 32 个源点。随后，算法对每个源点运行一次 Brandes 单源最短路径依赖计算，并用 32 个源点贡献的平均值给候选边打分，最后删除分数最高的边。它的真实计算成本清楚固定，为每步 32 次单源遍历。在 prefix smoke v2 中，`fixed_k32` 取得最低 active AUC 和最高 40 步 GCC 下降量，说明在当前实验里，它是后续改进方法必须超过的核心动态基线。
